NDU Manual GUI Test 自动化思路整理 -- Playwright or Selenium

Created2025-11-25|Updated2025-11-25|Testing & Automation
|Post Views:

当前手工执行流程

  1. Lease 存储设备

  2. 获取并确认存储信息：查找存储设备的 MGMT IP；登录 GUI 界面，确认当前存储版本信息

  3. 准备配套 Linux 环境：登录与该存储配套的 Linux 设备；确认是否与存储处于同一 VLAN；配置桌面环境，安装 Firefox

  4. 下载升级安装包：使用 wget 下载最新安装包

  5. 通过 VNC 进行图形化操作：使用 VNC 登录 Linux 桌面；通过 Firefox 访问存储 MGMT IP；GUI中上传最新安装包

  6. 执行升级操作（NDU）

  7. After NDU：能够升级完成；升级完成后是否存在异常告警信息；components的接入正常

为何需要考虑自动化？

虽然频率是两个月一次，不算高频，但流程长、串步骤多，人工操作很耗时间；并且容易因为环境差异，准备环境时需要检查很多的配置，过于繁琐；无法做到“到点自动触发”，还需要人盯日程和版本节奏

工具方案对比：Selenium vs Playwright

目前主要考虑两条路线：Selenium 和 Playwright

Selenium 现状

在现有用例中，Selenium 已经实现了部分 Configuration 的自动化配置

但脚本维护成本较高，整体可维护性并不理想

Playwright 的优势

原生支持自动等待机制，极大减少显式等待、轮询这种“脚本噪音”

稳定性在实践和社区反馈中普遍优于 Selenium

对现代 Web 应用（尤其是前端异步加载较多的界面）更友好

综合考虑当前脚本维护情况和后续扩展性，最终决定采用 Playwright 作为 NDU GUI test 的实现工具，并在此基础上进一步探索自动化触发与环境准备的简化方案
